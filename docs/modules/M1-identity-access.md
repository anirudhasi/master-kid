# M1 — Identity & Access (Module Spec)

**Status:** Draft for sign-off · **Stage:** 1 · **Owner:** platform
**Grounding:** based on actual code — `authService.ts` (provider seam, keep), migration
`001_tenancy.sql` (accounts table, extend), `lib/adminAuth.ts` (retire — security hole).

---

## 1. Responsibility

The single authority for: **who you are** (authentication), **what hat you wear** (roles),
and **what you may touch** (authorization). Every other module asks M1; none re-implement
role logic. This module is the root of the dependency graph — get it right once.

## 2. Actor model (the big decision in this spec)

Five actors. Two are *login identities*; the rest derive from them:

| Actor | Identity type | Notes |
|---|---|---|
| Parent | `accounts` row, role `parent` | Super-user; owns children; one phone = one login |
| Coach/Tutor | `accounts` row, role `coach` | Own login; attaches to children via handshake |
| **School** | `accounts` row, role `school` + `school_orgs` row | **NEW.** The account is the school administrator; the org record holds institution data. Staff seats come later (M5) |
| Admin | `accounts` row, role `admin`, **provisioned only via SQL/service-role** — never self-signup | See §5 security fix |
| Child | **NOT an account.** A `children` row + a **child-mode session** inside the parent's login | Deliberate: no independent child credentials = smaller DPDP/COPPA surface, no under-13 login consent problem. Revisit only when teens (Class 9+) need own devices — that gets its own ADR |

**Multi-role:** one human can be both parent and coach. Model as `role` (primary) +
`roles text[]` (all granted). UI offers a role switcher when `roles.length > 1`.

### Schema migration `012_identity_v2.sql`
```sql
alter table accounts drop constraint accounts_role_check;
alter table accounts add constraint accounts_role_check
  check (role in ('parent','coach','school','admin'));
alter table accounts add column if not exists roles text[] not null default array['parent'];
alter table accounts add column if not exists status text not null default 'active'
  check (status in ('active','disabled'));   -- M10 admin enable/disable acts here
update accounts set roles = array[role];
```
Backward-compatible: existing rows keep working; old code reading `role` is unaffected.

## 3. Authorization — central RBAC contract

One function, used everywhere (UI gating AND server-side enforcement):

```ts
// modules/identity/contracts.ts
export type Role = 'parent' | 'coach' | 'school' | 'admin'
export type Action = 'read' | 'create' | 'update' | 'delete' | 'manage'
export type Resource =
  | { kind: 'child'; childId: string }
  | { kind: 'course'; courseId: string }
  | { kind: 'post'; postId: string }
  | { kind: 'module'; name: ModuleName }        // module-level gating (M10 toggles)
  // extend per module as specs land

export interface IdentityContract {
  getSession(): Promise<Session | null>
  can(action: Action, resource: Resource): Promise<boolean>
  grantHandshake(childId: string): Promise<{ token: string; expiresAt: string }>
  redeemHandshake(token: string): Promise<{ ok: boolean; childId?: string }>
  revokeAccess(childId: string, granteeAccountId: string): Promise<void>
}
```

**Defense in depth — three layers, all active:**
1. **UI** calls `can()` to hide/disable what you can't do (UX only, zero security value).
2. **RLS** in Postgres is the real enforcement for browser→Supabase CRUD.
3. **`/api/*` Functions** re-check roles server-side for anything using the service-role key.

Policy matrix (authoritative; RLS policies implement it):

| | own account | own children's data | enrolled child's data | any account | platform config |
|---|---|---|---|---|---|
| parent | RW | RW | — | — | — |
| coach | RW | — | R (+ W on own sessions/milestones) | — | — |
| school | RW | — | R for rostered children (M5 defines write) | — | — |
| admin | RW | — | — | RW via server functions only | RW |

## 4. Authentication flows

- **Parent/Coach/School:** keep the existing `authService` provider seam (it's already
  well-designed). Phone OTP + Google OAuth. School signup adds an org-details step and
  lands in role `school`.
- **Child mode:** parent taps child → app enters child shell (`child.selected` event).
  Optional PIN lock to exit child mode (parent-set, stored hashed server-side) so a child
  can't wander into parent/billing screens. No separate JWT — same session, UI-scoped.
- **Handshake (coach↔child, later school↔child):** short-lived (72h), single-use, 8-char
  token generated by parent, redeemed by coach; creates an `enrollments` row (M4's table —
  M1 only brokers the grant). Revocation deletes/deactivates the enrollment; RLS reflects
  it immediately.

## 5. SECURITY FIX — server-side admin auth (blocks public launch)

**Current hole:** `lib/adminAuth.ts` verifies an admin password in the browser against a
SHA-256 hash shipped in the JS bundle via `VITE_` env. Anyone can extract the hash and
crack it offline; unsalted SHA-256 offers ~zero resistance. Also: role escalation happens
client-side.

**Fix (part of M1 build, non-negotiable before launch):**
1. Delete `lib/adminAuth.ts` and all `VITE_ADMIN_*` vars.
2. Admin accounts are normal Supabase auth users whose `accounts.role='admin'` is set
   **only** via service-role (SQL / provisioning script). No self-serve path.
3. All admin operations go through `/api/admin/*` SWA Functions which (a) verify the JWT,
   (b) check `role='admin'` server-side, (c) act with the service-role key, (d) write to
   `admin_audit_log` (migration 009 already exists — use it).
4. Add a second factor for admin (TOTP or email-code on top of OTP) — cheap, high value.

## 6. Events emitted

`account.created` · `account.role_changed` · `account.status_changed` ·
`handshake.granted` · `handshake.redeemed` · `access.revoked`

## 7. Owned data

`accounts` (extended), handshake tokens table (new, hashed at rest), child-mode PIN
(hashed, server-side). NOT owned: `children` (M2), `enrollments` (M4) — M1 brokers
grants but the domain modules own their records.
